# Poll Type Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a `pollType` field to polls — `DATE` (options are datetimes, current behaviour) or `TEXT` (options are arbitrary text strings).

**Architecture:** Add `PollType` enum + `proposedOptions: string[]` to the Mongoose model. DATE polls continue using `proposedDates`; TEXT polls use `proposedOptions`. The vote index mechanism (`dateIndex`) is type-agnostic. `computeWinnerIndex` and vote validation are updated to use the correct options array per type. The client switches inputs and display formatting based on `pollType`. Event creation from a winner is only available for DATE polls. Existing polls without `pollType` default to `DATE` via the `!pollType || pollType === 'DATE'` guard everywhere.

**Tech Stack:** Node.js / Express / Mongoose / TypeScript (server), React 19 / TypeScript / Vite (client)

- [ ] **Step 1: Fix `computeWinnerIndex` to use the right options array**

The helper currently uses `poll.proposedDates.length` as the index space. Replace it so it works for both types:

```typescript
function computeWinnerIndex(poll: any): number {
  const optionCount = poll.pollType === PollType.TEXT
    ? (poll.proposedOptions?.length || 0)
    : poll.proposedDates.length;

  const scores = Array.from({ length: optionCount }, (_, i) => {
    const yes = poll.responses.filter((r: any) =>
      r.votes.find((v: any) => v.dateIndex === i && v.answer === 'YES')
    ).length;
    const maybe = poll.responses.filter((r: any) =>
      r.votes.find((v: any) => v.dateIndex === i && v.answer === 'MAYBE')
    ).length;
    return { index: i, yes, total: yes + maybe };
  });

  scores.sort((a, b) =>
    b.yes !== a.yes ? b.yes - a.yes : b.total !== a.total ? b.total - a.total : a.index - b.index
  );
  return scores[0].index;
}
```
